    Now, monologue aside, what even is CSS?

    CSS Is. . .

    CSS, or Cascading Style Sheets, is a markup language used to apply styles to HTML.

    Generally, CSS is responsible for controlling the layout, color, font, and visual appearance of pages without altering your HTML document. As a result, CSS allows for more flexible and maintainable websites as a whole.

    If I were to explain CSS as simply as I possibly can: think of a webpage as a house. If HTML is the structure, then think of CSS as the interior design and decoration. Essentially, CSS makes your websites look good before anything else.

    The Three Main Ways to Use CSS

    CSS is used in three main ways:

    • Inline CSS
    • Internal CSS
    • External CSS

    First, let's begin with inline CSS.

    Inline CSS is written within the HTML element you wish to design. For example, let's say you want to change a paragraph element's color:

    <p style="color: red;">This is red text</p>

    This style of CSS can seem convenient at first when you want to style individual elements, but this can lead to issues with code readability and maintainability later down the line. It is best practice to use inline CSS when testing a style on an element.

    Now, let's move on to internal CSS.

    Internal CSS is also written within an HTML document, but you don't write the CSS code within the HTML element. Instead, you write your CSS code in the head section of your HTML document with the <style> tag. It will look a little something like:

    <style>
    p {
     color: red;
    }
    </style>

    This style of CSS is most useful when styling a single document.

    Now, for external CSS.

    External CSS is what you have probably seen the most throughout your time in making webpages. This style of CSS is written in a separate .css file and linked to the HTML document, located in the <head> section on top of the HTML document. The code to link your external .css file to your HTML document will look a little something like:

    <link rel="stylesheet" href="styles.css">

    Now, you've been shown multiple examples of CSS and their uses, but now we need to delve just a bit deeper into syntax. We need to take a closer look at how CSS rules actually work.

    Basic Syntax and Rules

    For starters, a CSS rule is composed of two main parts:
    A selector block and a declaration block

    You saw a version of this earlier in the examples of CSS use, but here is the basic syntax of a CSS rule.

    selector {
     property: value;
    }

    You have your selector, the property you want to set, and the value you want to set it to. Think about it this way:

    • Selector — targets the specific HTML element you want to style. Selectors can include type, .class, and #id
    • Property — targets what you want to change about the HTML element. This can range from color, to font-family, to background, and much more.
    • Value — sets the value of the property you wish to style.

    You can also apply the same set of styles to multiple selectors. Just look at this code for example.

    #title,
    .subtitle {
     color: blue;
    }

    Here, you can see that there are multiple selectors. These selectors target elements in your HTML document that have an id value (beginning with #) of "title", while also targeting elements that have a class value (beginning with .) of "subtitle".

    And, please, please, PLEASE don't forget your semicolons. They may seem little and unimportant, but they separate your CSS declarations, and if they are forgotten it can cause errors in your CSS. Thankfully, code editors like Visual Studio Code are very good at letting you know when you forget them.

    I may, or may not, be speaking from experience. Don't ask.

    Selectors

    You will learn more about selectors in detail in the Attribute Selectors portion of this guide. But for now, all you need to know is that in the early stages of your CSS journey you will mainly deal with three types of selectors:

    Element Selectors

    Element selectors target all instances of an HTML tag.

    p {
     line-height: 1.5;
    }

    Class Selectors (.)

    Class selectors target all elements with a specific class. This will be your workhorse in CSS.

    .warning {
     background-color: red;
    }

    One class can be used everywhere, and an element can have multiple classes.

    ID Selectors

    ID selectors target a single, unique element by its unique ID.

    #title {
     color: green;
    }

    Be sure to use IDs sparingly. They are powerful, but do not have the flexibility of classes. As a result, they become very hard to override when needed.

    Selector Example When to Use
    Element p Styling ALL paragraphs the same way.
    Class .warning Reusble styles for all elements with a given class.
    ID #title For one of a kind elements.

    And that's essentially it, honestly. You can build entire websites with just these three.

    The Cascade Algorithm

    The cascade algorithm can seem a bit complex at face value, and it can become complex. But it is very important to know the basics, at least. You can learn more about this in the Attribute Selectors section of this guide, but let's begin with the basics first.

    As we know, CSS stands for Cascading Style Sheets. The Cascading portion is actually an algorithm your browser uses to decide what style to use when multiple rules target the same element.

    Specificity

    Specificity acts as a "weight", and some selectors are "heavier" than others.

    Here's a list of the selectors from strongest specificity to weakest

    1. Inline Styles (style="...")
    2. ID Selectors (#title)
    3. Class Selectors (.warning)
    4. Element Selectors (p)

    Source Order

    If the specificity is tied, the last rule that targets the same element wins.

    For example:

    p { color: red; }
    p { color: blue; } /* This wins */

    Inheritance

    Some properties pass from parent to child. For example, if you had a p element inside of a div element.

    <div>
    <p>This will inherit gray from the div</p>
    </div>

    And then styled the div:

    div { color: gray; }

    Then the p element would inherit the color rule of the div. It is important to know that inheritance doesn't apply to everything. For example, margins, padding, and borders generally are not inhrited, but colors and fonts are inherited.

    !important: The Gamebreaker

    !important will force whatever element, regardless of initial specificity, to have the utmost highest specificity.

    Do not use !important unless absolutely necessary. This breaks the natural cascade, and if you eventually need to override it you will have to slap the !important tag on something else with an even higher specificity, and it becomes an agonizingly endless cycle of importance. Use it wisely, unless you enjoy hours of debugging in CSS.

    p { color: red; !important } /* Forces this to win */

    Cascade in Practice

    Take the following HTML:

    <p id="content" class="text">What color is this?</p>

    And then take this CSS stylesheet

    /* Element selector — weakest */
    p {
     color: gray;
    }

    /* Class selector — stronger */
    .text {
     color: green;
    }

    /* ID selector — strongest */
    #content {
     color: blue;
    }

    If you remove the ID, green wins (class beats element). If you remove both ID and class, gray wins (element, and it's the only rule left).

    And that's the cascade. Nothig more than weight and order. We will go a bit further in depth regarding this later, but this is it for now.

    Comments

    Now, I know this is a strange transition, going from basic syntax, to selectors, to cascade, to... comments? Just hear me out, okay?

    We know how to make code now, at least we have an idea. But, what about future you? Or, as you grow in your journey as a web developer, what about other people? This is where the value of comments is the highest. You aren't just coding for yourself today, but you are also coding for yourself tomorrow, in a week from now, even months or years from now.

    I say this because it is important to know not just what you're doing, but why you're doing it. This is where comments come in.

    Comments are essentially notes you can leave in your code that are completely ignored by the browser, and as a result are not shown in your "final prouct". Web developers, use comments to explain the purpose of code, or to separate sections of the code. And, in CSS, they look a little something like this:
    /*changes color of title for the hover easter egg*/
    #title:hover {
     color: blue;
    }

    Take a look at the text in the top line of the code. That is a comment. Comments begin with /* and end with */. Comments in CSS can also span multiple lines:
    /*changes color of title when hovered
    subject to change*/
    #title:hover {
     color: blue;
    }

    Now, with comments, they should be used to explain why the code does what it does, rather than what the code does. Your code should already explain what it does on its own, with proper naming.

    /* Avoid: states the obvious */ p {
     color: red; /* makes text red */
    }

    /* Better: explains reasoning */
    p {
     color: red; /* matches brand error color from style guide */
    }

    Comments can also be used for separating sections of your stylesheet:

    /* ========== TYPOGRAPHY ========== */
    /* ========== LAYOUT ========== */
    /* ========== BUTTONS ========== */

    For commenting out code to disable it, usually to help with debugging:

    /* Temporarily disabled to test layout issue */
    /*.sidebar {
     float: right;
     width: 30%;
    } */

    And to-do notes for future work, down the line:
    /* TODO: Add hover state for mobile */
    /* FIXME: Button alignment breaks in Safari */

    Also, keep in mind that you can not nest comments in CSS. The moment your comment has a */, it will close the comment.

    /* This comment/* will break */ when you try to nest */

    CSS Reset

    Browsers have default styles. They add margins to body, padding to ul, boldness to headings. It essentially tries to do its own thing. These defaults are inconsistent across browsers and interfere with your designs.

    So, we can reset it with the universal selector (*).

    This is the simplest CSS reset you'll usually find:

     * {
    margin: 0;
    padding: 0;
    box-sizing: border-box;
    }

    Be sure to put this on top of your stylesheet, before any other rules.

    Browser Developer Tools

    Every modern browser that you can think of has neat developer tools that allow you to inspect and modify CSS in real time. You may know this tool as "Inspect Element", or just "Inspect".
    As a web developer, this will become your best friend. Well, this and MDN web docs. It's a tight top two.

    Now, how to use it. If you're on Windows or Linux, You can hit the f12 key (or fn + f12 on laptops).
    And for Mac, just hit Cmd + Option + I.
    And, if all else fails, you can always right click and select the option to inspect.

    Here's some of what you can do with DevTools:

    • See exactly which styles are applying to an element
    • See which styles are crossed out (overridden by specificity or order)
    • Toggle properties on and off
    • Test changes live without editing your files

    So, if something isn't working, just open DevTools. It shows you exactly what the browser is seeing, and not just what you intended with your code.

    What You Need to Remember

    1. Use external CSS — One file, linked from HTML
    2. Remember your syntax — selector { property: value; }
    3. Use classes — They are maintainable and flexible
    4. Understand specificity — ID > Class > Element
    5. Source order matters — Last rule wins if specificity is tied
    6. Avoid !important — It's a trap!
    7. Be sure to reset CSS — Start from a clean slate
    8. Use DevTools — When it doubt, inspect it out

    Attribute Selectors

    We have covered selectors briefly in the Basics section of this guide. But, it is important to understand selectors and the cascade in detail.

    Selectors and Why They Matter

    Selectors are how you tell CSS what to target. We touched on this very briefly in the basic syntax section, and now it's time to go into a bit more detail.

    You will be working with four selectors constantly throughout your time in CSS:

    Universal Selectors (*)

    The universal selector targets every element on the page.

    * {
     margin: 0;
     padding: 0;
    }

    Universal selectors are mostly used as CSS resets to remove the browser's default spacing. Be sure to use this sparingly, as applying styles to every element can be taxing on performance, and overly broad.

    Element Selector

    Element selectors target all instances with a specifc HTML tag. For example, if you want to target all p elements in your HTML file, your CSS would look like:

    p {
    line-height: 1.5 rem;
    color: purple; }

    With this CSS code, every single p element will be targeted with the given CSS rule.

    Class Selector (.)

    Class selectors target elements with a specific class attribute. This selector is most commonly used, and the most flexible.

    .warning {
    color: yellow;
    }

    A single class can be reused across multiple elements, and a single element can have a multitude of classes.

    ID Selector (#)

    ID selectors target a single element with a specific id attribute. IDs must be unique within a page. You will never see the same id attribute twice in the same page.

    #title {
    background-color: navy;
    color: white;
    }

    IDs have the highest specificity of the four selectors (more on that later), so use them sparingly. Use them for unique elements or sections like headers, footers, or main containers.

    The Cascade Algorithm

    Now, as we touched on at the beginning of this guide, CSS stands for (say it with me now):
    Cascading. Style. Sheets.

    Now what does that Cascading part mean?

    The cascading algorithm determines how the browser decides which styles to apply when multiple rules target the same element.

    Well, when the browser renders your page, it will follow a four stage algorithm:

    Stage One: Origin

    Your browser will collect styles from three origins:

    • Browser defaults—the user agent stylsheet.
    • User styles—usually the accessibility settings and browser extensions.
    • Author styles—this is your CSS!

    Now, in terms of precendence, the author's stylesheet will take precedence over the user styles, which take precedence over the browser defaults.

    Stage Two: Importance

    Now, moving on to importance, it is... important (pun intended) to know that the browser goes within each origin and separates the rules marked with !important before sorting the rules by importance. That's a lot of important being said, so hear me out. Think of importance as a two-tier system:

    The !important tier and the normal tier.

    The browser will go through the three origins we discussed earlier (browser defaults, user styles, author styles) and essentially pluck the rules marked with !important and put them in the !important tier. After the !important rules are separated, the browser will begin to sort. Here is the order of importance from highest to lowest:

    1. Author's !important rules
    2. User's !important rules
    3. Browser's !important rules
    4. Browser's normal rules
    5. User's normal rules
    6. Author's normal rules

    Now, I know you see it. And, I'm sure you might be thinking
    "ADJ, why is the author's normal rules so low, especially when you said that the author's stylesheet takes precedent earlier?"

    First of all, I am so glad you've been paying attention. Really good job. I'm serious. But, also, the author's normal rules take lowest priority for the sake of the user's ability to override styles for accessibility needs.

    Makes sense? Okay. Let's move on to stage three.

    Stage Three: Specificity

    Alright, so the browser has filtered by origin and importance. Now, the browser calculates specificity.

    Think of specificity as a weight. Every kind of selector has a weight. When two or more CSS rules target the same element, the browser applies the rule with the highest specificity weight, regardless of the order they are in.

    With this in mind, specificity can become a hierarchical scoring system where every selector type contributes points, but not all points are equal. Take this table in mind:

    Level Selector Type Example Score
    1 (highest) Inline styles style="color: red;" 1,0,0,0
    2 ID selectors #header 0,1,0,0
    3 Psuedo-classes :hover 0,0,1,0
    3 Attribute selectors [type="text"] 0,0,1,0
    3 Class selectors .warning 0,0,1,0
    4 (lowest) Element selectors p, h1 0,0,0,1

    The universal selector (*), combinators (+, >, ~,   ), and the :where() pseudo-class do not contribute to specificity.

    Now that we have our table, and know what each selector weighs, we can use this to tackle more complex selectors. Take the following selector for example:

    #sidebar .nav-links a:hover {
    }

    I know it looks strange and confusing, but let's work it out.

    First, let's break it down into four parts.

    Part Selector Type Specificity
    #sidebar ID 0,1,0,0
    .nav-links Class 0,0,1,0
    a Element 0,0,0,1
    :hover Pseudo-class 0,0,1,0

    Add the specificities column together: 0,1,2,1.

    Now, in terms of specificity, the browser goes from the leftmost number to the right. In this case, it would go from 0, to 1, to 2, to 1 and compare it to the numbers of other elements.

    For example, let's have the browser compare this sample element to .nav-links a:hover{}.

    #sidebar .nav-links a:hover {} /*0,1,2,1*/
    .nav-links a:hover{} /*0,0,2,1*/

    Let's break this into a table as well.

    Element Column 1 Column 2 Column 3 Column 4
    #sidebar .nav-links a:hover {} 0 1 2 1
    .nav-links a:hover {} 0 0 2 1

    Now, that the elements are broken down into columns, we can compare each element's value in each column. Moving from left to right, if a the elements have an equal value in said column you move on to the next. The first column where one number is higher determines the winner.

    In our example, #sidebar .nav-links a:hover {} wins because its column 2 value of 1 is greater than 0.

    In terms of practicality, it is best practice to use classes. They have enough specificity to override element selectors, but can be overridable by ID selectors if needed.

    When it comes to specificity, always remember the rule of thumb: make your specificity as high as you need it to be, but no higher than that.

    Stage Four: Source Order

    Now, if the browser comes across two rules that both:

    1. Have the same origin
    2. Have the same importance
    3. Have the same specificity

    The browser will then look at source order. The rule that appears later in the stylesheet wins.

    Example:

    p { color: red;}
    p { color: blue;} /* This will be the winner */

    Put it All Together

    Take this HTML code for example:

    <p id="content" class="text">What color am I?</p>

    And then this stylesheet:

    /* Rule A: specificity 0,0,0,1 */
    p { color: gray; } /*
    Rule B: specificity 0,0,1,0 */
    .text { color: green; }
    /* Rule C: specificity 0,1,0,0 */
    #content { color: blue; } /*
    Rule D: same selector as Rule C, appears later */
    #content { color: purple; }

    This is how the browser would break it down into the cascade algorithm:

    1. Origin: All author normal rules
    2. Importance: No rules marked as !important
    3. Specificity: Rules C and D will win (0,1,0,0 vs lower values)
    4. Source Order: Rule D appears after Rule C, so Rule D wins.

    As a result of Rule D winning, the text will be purple.
